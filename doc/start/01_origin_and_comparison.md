# PQNodium — 项目起源与对比

## 项目背景

PQNodium 起源于之前的 **ZFeiQ** 项目（一个基于 IPMSG 协议的局域网 P2P 即时通讯系统，Python/PyQt5 实现，具备 X25519/ChaCha20 加密、NPU 加速 OCR、文件传输等功能）。

但在技术选型和功能规划过程中，发现新项目的技术栈、架构、安全模型与 ZFeiQ 存在**代差级差异**，本质上是一个全新项目，因此重新命名并重新规划。

## ZFeiQ vs PQNodium 对比

| 维度 | ZFeiQ (已归档) | PQNodium (规划中) |
|------|---------------|------------------|
| **项目定位** | 局域网 IPMSG 替代 | 全场景去中心化聊天 |
| **网络范围** | 仅 LAN (UDP 广播) | LAN + WAN (DHT + NAT 穿透) |
| **架构** | 纯 P2P 广播 | 混合 P2P (DHT 发现 + QUIC 直连 + Relay fallback) |
| **语言** | Python 3.8+ | Rust |
| **GUI** | PyQt5 | Tauri v2 + React + Tailwind |
| **网络协议** | 原始 UDP/TCP (端口 2425) | libp2p + QUIC (UDP-based, 0-RTT) |
| **传输层加密** | 无 (依赖应用层) | QUIC 内置 TLS 1.3 |
| **应用层加密** | Noise (X25519 + ChaCha20-Poly1305) | Noise (X25519 + **ML-KEM-768 混合**) |
| **身份签名** | 无 (MITM 漏洞已记录) | Ed25519 + **ML-DSA-65 混合** |
| **密钥交换** | X25519 ECDH | **X25519 + ML-KEM-768** (PQ/T hybrid) |
| **后量子支持** | ❌ | ✅ (FIPS 203/204 标准) |
| **节点发现** | UDP 局域网广播 | Kademlia DHT + 邀请码 (WAN) |
| **NAT 穿透** | ❌ (仅限同网段) | ✅ (UDP hole punching + QUIC + Relay fallback) |
| **消息路由** | 直接点对点 | DHT 路由 + GossipSub (群组) |
| **消息协议** | IPMSG 格式 | 自定义 / MLS (RFC 9420) |
| **群组消息** | 基础组广播 | MLS TreeKEM (前向安全 + 后向安全) |
| **前向保密** | ✅ (临时密钥) | ✅ + 后量子保密 |
| **文件传输** | TCP 直接传输 | libp2p stream + 可选 CID 分片 |
| **OCR / AI** | ✅ PPOCRv4 (RKNN NPU) | 待定 (可复用) |
| **平台支持** | Linux (RK3566) + Windows | Windows / macOS / Linux / Android (目标) |
| **多语言 (i18n)** | ✅ | 待定 |
| **CLI** | ✅ (prompt_toolkit) | ✅ (计划) |
| **截图功能** | ✅ (跨平台) | 待定 |
| **安全审计** | 文档记录 MITM 漏洞 | 设计阶段即考虑 PQC + hybrid 签名 |
| **Harvest-Now-Decrypt-Later 防护** | ❌ | ✅ (ML-KEM) |
| **开发状态** | 已归档 | Phase 3 已完成，Phase 4+ 规划中 |

## 核心差异总结

| 方面 | ZFeiQ | PQNodium | 差距 |
|------|-------|----------|------|
| **可达性** | 同网段设备 | 全球任意节点 | 质变 |
| **抗量子** | 无 | ML-KEM + ML-DSA | 代差 |
| **安全性** | 有已知 MITM 漏洞 | Hybrid 签名防御 | 代差 |
| **工程** | Python (GIL 限制) | Rust (内存安全+并发) | 代差 |
| **可扩展** | 广播模式，节点多即拥堵 | DHT + GossipSub，水平扩展 | 质变 |
