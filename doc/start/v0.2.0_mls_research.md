# v0.2.0 Phase 1 — MLS 选型调研报告

> 调研日期: 2026-05-15
> 目标: 评估 Rust MLS (RFC 9420) 实现，为 PQNodium v0.2.0 群组加密选型

## 1. MLS 协议概述

MLS (Messaging Layer Security, RFC 9420) 是 IETF 标准化的异步群组加密协议：

| 特性 | 说明 |
|------|------|
| **密钥管理** | Ratchet Tree — O(log N) 密钥更新效率 |
| **前向保密 (FS)** | 每 Commit 创建新 epoch，旧密钥删除 |
| **后妥协安全 (PCS)** | Update 操作恢复被妥协的树不变性 |
| **元数据保护** | Sealed Sender — 发送者身份在 PrivateMessage 中加密 |
| **标准状态** | RFC 9420 (2024-07 发布)，PQC cipher suite 在 draft-ietf-mls-pq-ciphersuites-04 |

### 核心概念

- **KeyPackage**: 预发布凭证（类似 Signal prekey），包含 HPKE init key + 签名密钥 + 能力声明
- **Epoch**: 组状态的线性版本，每次 Commit 创建新 epoch，独立密钥
- **Ratchet Tree**: 完美二叉树，叶子=成员，中间节点=HPKE 密钥对，UpdatePath 沿 filtered direct path 更新
- **Commit**: 组状态转换操作（执行 Add/Remove/Update proposals + 可选 UpdatePath）
- **Welcome**: 加密的组状态邀请消息，用新成员 KeyPackage 中的 HPKE init key 加密

### 与 PQNodium 现有架构的映射

| PQNodium 现有 | MLS 对应 | 兼容性 |
|--------------|---------|--------|
| `HybridKem<X25519Kem, MlKem768Kem>` | HPKE KEM (ML-KEM-768+X25519 hybrid) | 直接对应 |
| `HybridSignature<Ed25519, MlDsa65>` | MLS Signature (ML-DSA-65+Ed25519) | 直接对应 |
| `ChaCha20Poly1305Cipher` | MLS AEAD | 直接对应 |
| `Identity` (Ed25519+ML-DSA-65) | MLS LeafNode credential | 需适配 |
| `PeerId` (SHA-256 hash) | MLS KeyPackage identity | 概念不同 |
| Gossipsub broadcast | MLS Delivery Service | 可行，需缓冲层处理乱序 |
| 1-to-1 handshake (2-round) | MLS Welcome + Commit | 完全不同模型 |

## 6. 关键决策因素

### 6.1 PQNodium 的特殊需求

1. **必须使用 ML-KEM-768 + X25519 hybrid KEM** — 这是 PQNodium 的核心安全承诺
2. **纯 Rust，无 C 依赖** — 保持跨平台构建简洁
3. **与 Gossipsub 集成** — MLS 需要在无序广播网络上运行
4. **P2P 架构** — MLS 设计假设有 Delivery Service (服务器)，PQNodium 没有

### 6.2 MLS 的架构冲突

MLS 假设：
- 有序消息投递（epoch 必须递增）
- Delivery Service 做消息分发
- KeyPackage 目录服务

PQNodium 现实：
- Gossipsub 是无序广播
- 无中心化服务器
- 节点通过 Kademlia DHT 发现

这意味着即使选择了 MLS 库，仍需构建**适配层**处理：
- 乱序 Commit 缓冲
- Welcome 消息的点对点投递
- KeyPackage 的 DHT 发布/查询

## 7. 选型建议

### 保守评估结论

**两个库都不满足"可直接采用"的标准：**

- OpenMLS: PQC 支持未合并 (PR #1994 Open)，API 不稳定
- mls-rs: PQC 需要 AWS-LC unstable，维护者流失严重

### 推荐路径: 三阶段渐进策略

#### 阶段 A: 轻量级组加密 (可立即实施)

**不引入 MLS 库**，基于 PQNodium 现有 handshake 扩展：

```
组创建者:
  1. 生成组密钥 group_key (随机 32 bytes)
  2. 用每个成员的 HybridKem 封装 group_key → group_key_ct[member_i]
  3. 广播 GroupSetup 消息 (Gossipsub)

组成员:
  1. 收到 GroupSetup → HybridKem 解封 group_key
  2. 用 group_key 派生 send_key = KDF(group_key, "send-" || my_index)
  3. 发送消息: AEAD Seal(send_key, nonce, plaintext, aad)

密钥更新:
  - 定期重新生成 group_key
  - 新 group_key 用当前 key 加密广播
```

优势: 零外部依赖，完全复用现有 `pqnodium-core` crypto，快速实现
局限: 无 FS (group_key 泄露=全部泄露)，无 PCS，非标准协议

#### 阶段 B: 等待 OpenMLS PQC 合并

监控 OpenMLS PR #1994 合并状态。合并后：
- 使用 `ml-kem` (RustCrypto) 后端 — PQNodium 已有依赖
- 选择 `MLS_128_MLKEM768X25519_AES128GCM_SHA256_Ed25519` cipher suite
- 实现 Gossipsub → MLS Delivery Service 适配层

预计时间线: 2026 Q3-Q4 (PR 24 files, 843 additions, 活跃 review 中)

#### 阶段 C: 完整 MLS 集成

基于阶段 B 的经验，实现完整 MLS：
- TreeKEM ratchet tree
- 完整 FS + PCS
- Sealed Sender
- RFC 9420 合规
- IETF interop test vectors 验证

### 如果必须立即选择一个库

**选择 OpenMLS**，理由：
1. 社区更大、更活跃 (4x stars, 2.5x 贡献者)
2. 维护者更稳定 (专业密码学公司 vs 流失的 AWS 团队)
3. 文档更完善 (book + blog + validation dashboard)
4. PQC PR 使用 RustCrypto `ml-kem` — 与 PQNodium 已有依赖一致
5. 即使 PR #1994 未合并，也能先使用标准 cipher suite (X25519) 作为降级方案

## 8. IETF PQC + MLS 标准跟踪

| 草案 | 状态 | 对 PQNodium 的影响 |
|------|------|-------------------|
| draft-ietf-mls-pq-ciphersuites-04 | Active | 定义 ML-KEM-768+X25519 hybrid cipher suite，与 HybridKem 一致 |
| draft-ietf-mls-combiner-02 | Active | 同时运行传统+PQ MLS 会话，amortized PQ 方案 |
| draft-ietf-hpke-pq-04 | Active | HPKE 添加 PQ KEM 支持 |

PQNodium 的 `HybridKem<X25519Kem, MlKem768Kem>` 构造 `shared_secret = SHA-256(classic_ss || pqc_ss)` 与 draft-ietf-mls-pq-ciphersuites 中的 hybrid KEM 高度一致。

## 9. libp2p MLS 支持

**libp2p 没有内置 MLS 支持**，也没有相关计划。MLS 应作为应用层协议在 Gossipsub 之上运行。PQNodium 需要自行实现：
- MLS 消息封装在 Envelope payload 中
- Welcome 消息通过 libp2p direct message 点对点投递
- Commit/Proposal 通过 Gossipsub 广播
- 乱序 Commit 本地缓冲排序
