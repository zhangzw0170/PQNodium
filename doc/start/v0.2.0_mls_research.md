## 1. MLS 协议概述 (RFC 9420)

MLS (Messaging Layer Security) 是 IETF 标准化的异步群组加密协议。

### 核心概念

| 概念 | 说明 |
|------|------|
| **KeyPackage** | 预发布凭证 (类似 Signal prekey)，含 HPKE init key + 签名密钥 |
| **Epoch** | 组状态版本，每次 Commit 创建新 epoch + 独立密钥 |
| **Ratchet Tree** | 完美二叉树，叶子=成员，O(log N) 密钥更新 |
| **Commit** | 组状态转换 (执行 Add/Remove/Update + UpdatePath) |
| **Welcome** | 加密组状态邀请，用新成员的 HPKE init key 加密 |
| **TreeKEM** | 树密钥交换协议，从 epoch_secret 派生 8 个子密钥 |

### 与 PQNodium 的映射

| PQNodium | MLS | 兼容性 |
|----------|-----|--------|
| `HybridKem<X25519Kem, MlKem768Kem>` | HPKE KEM (ML-KEM-768+X25519) | 直接对应 |
| `HybridSignature<Ed25519, MlDsa65>` | MLS Signature | 直接对应 |
| `ChaCha20Poly1305Cipher` | MLS AEAD | 直接对应 |
| Gossipsub broadcast | MLS Delivery Service | 需缓冲层 (乱序) |
| 无中心服务器 | MLS 假设有 Delivery Service | 架构冲突 |

### MLS 的架构冲突 (关键问题)

MLS **假设**：
- 有序消息投递 (epoch 必须递增)
- Delivery Service 做消息分发
- KeyPackage 目录服务

PQNodium **现实**：
- Gossipsub 是无序广播
- 无中心化服务器
- 节点通过 Kademlia DHT 发现

> 这正是 p2panda 团队拒绝使用 MLS 的原因：**MLS 不适合纯 P2P 架构**。

## 2. 所有候选方案评估

### 2.1 自建 Sender Key + HybridKem 分发 ⭐ 推荐

**思路**: 参考 Signal Sender Keys / Matrix Megolm 的设计模式，用 PQNodium 已有的 `HybridKem` 分发组密钥。

```
组创建者:
  1. 生成随机对称组密钥 group_key (32 bytes)
  2. 对每个成员: HybridKem.encapsulate(member_pk) → group_key_ct[i]
  3. 广播 GroupSetup { members[], encrypted_keys[], sender_key }

组成员:
  1. HybridKem.decapsulate(ct[i]) → group_key
  2. 发送消息: AEAD.Seal(group_key, nonce++, plaintext, aad)

成员变更:
  - 新成员加入: 重新生成 group_key, 用新成员 pk 封装
  - 成员离开: 重新生成 group_key, 仅分发给剩余成员

进阶 (Sender Key 模式):
  - 每个成员有独立 sender_key + Chain Key ratchet
  - 发送: mk = H1(ck), ck = H2(ck) — 每消息唯一密钥
  - 接收: 用发送者公钥验证签名
```

| 维度 | 评估 |
|------|------|
| **新依赖** | 零 — 完全使用 pqnodium-core 已有的 HybridKem + ChaCha20Poly1305 |
| **PQC** | 原生 — X25519+ML-KEM-768 hybrid, 与现有 1:1 握手一致 |
| **前向保密 (FS)** | 部分 — 每消息 ratchet 提供消息级 FS，但 group_key 泄露=全部泄露 |
| **后妥协安全 (PCS)** | 弱 — 仅通过成员变更时的 re-key 实现 |
| **复杂度** | 低 — 约 500-800 行新代码 |
| **工期** | 2-3 周 |
| **带宽开销** | ML-KEM-768 密文 1088 bytes × N 成员 + 消息头 ≈ 10 KB (10 人组 re-key) |
| **升级路径** | 未来可无缝替换为 MLS 或 mKEM |

### 2.2 vodozemac (Matrix Megolm) + HybridKem

**思路**: 使用 Matrix 基金会的 `vodozemac` crate (Megolm 发送者密钥 ratchet)，通过 PQNodium HybridKem 分发 session key。

| 维度 | 评估 |
|------|------|
| **crate** | `vodozemac` 0.10.0 — Matrix 官方 Rust 实现，生产验证 |
| **新依赖** | 1 crate (vodozemac) + 传递依赖 (curve25519-dalek, ed25519-dalek, aes, sha2 等) |
| **PQC** | 需适配 — vodozemac 用 Curve25519，但 Megolm ratchet 本身是纯对称操作 (量子安全) |
| **FS** | 部分 — Megolm ratchet 前进，但不能回退。泄露=未来全部泄露 |
| **PCS** | 无 — 明确的安全权衡 (Matrix 文档承认) |
| **优势** | 生产验证 (Element X 客户端)，API 简洁，session pickling 支持 |
| **劣势** | 引入 curve25519-dalek (PQNodium 已有 x25519-dalek，可能冲突)，无 PQC 原生支持 |
| **工期** | 3-4 周 |

### 3.1 Cisco mlspp (已包含在 2.3)

唯一的开源 C/C++ MLS 实现。其他大型厂商 (Apple, Google) 的 MLS 实现均为闭源。

### 3.2 liboqs (Open Quantum Safe) — PQC 原语库

| 维度 | 评估 |
|------|------|
| **仓库** | open-quantum-safe/liboqs, 2,933 stars, C |
| **许可** | MIT |
| **Rust FFI** | 官方: `oqs` crate (safe wrapper) + `oqs-sys` |

2023 年宣布 MLS "on track"，但截至 2026 年初仍未发布。GPL-3.0 许可。不推荐。

### 3.4 BoringSSL / AWS-LC — 无群组加密

TLS 层密码库，仅有个体 PQC 原语 (ML-KEM)，无群组协议。

## 4. 学术前沿

### 4.1 mKEM (Multi-Recipient KEM) — 值得关注

**PQShield / NIST PQC Conference 2024**: mKyber 扩展 ML-KEM 到 N 个接收者，单次封装可分解为每人独立解封。

| 指标 | 传统 N×ML-KEM | mKEM |
|------|-------------|------|
| 10 人组 re-key 带宽 | ~10,680 bytes | ~1,184 bytes |
| 每增加 1 人 | +1,088 bytes | +48 bytes |
| 节省 | — | **9×** |

**状态**: 学术论文，无生产实现。可作为未来优化目标。

### 4.2 DCGKA (Decentralized CGKA) — 理论最优

Weidner et al. (2020) 证明 P2P 群组加密无需中心协调器。p2panda-encryption 是该论文的 Rust 实现。

**结论**: 理论基础扎实，但实现尚不成熟，无 PQC。

### 4.3 draft-ietf-lamps-cms-kyber-13 — 标准化的 "encrypt-to-multiple"

IETF 草案标准化了 ML-KEM 多接收者加密模式：生成对称密钥 → 为每个接收者分别封装 → 广播。

**状态**: 已到 draft-13，接近 RFC。与方案 2.1 的设计完全一致。

## 6. 推荐决策

### 推荐方案: Sender Key + HybridKem (方案 2.1)

**理由 (保守原则)**:

1. **零新依赖** — PQNodium 已有 ml-kem, x25519-dalek, chacha20poly1305, sha2 等所有必需原语
2. **PQC 原生** — HybridKem<X25519Kem, MlKem768Kem> 与 1:1 握手完全一致
3. **架构匹配** — Gossipsub 广播 + HybridKem 分发，无 Delivery Service 依赖
4. **已验证** — `pqnodium-core` 的 HybridKem 经过 194 tests + 安全审计
5. **可升级** — 未来可替换为 mKEM (9× 带宽优化) 或 MLS (完整 FS/PCS)

**已知局限 (可接受)**:
- 无后妥协安全 (PCS) — 成员离组后需 re-key
- 组密钥泄露=全部泄露 — 需定期轮换
- 大组带宽开销 — 每次 re-key O(N) HybridKem 封装

**实现规划 (v0.2.0 Phase 2-7)**:

```
Phase 2: GroupKey 模块 (pqnodium-core)
  - GroupKeyPair: 生成/封装/解封组密钥
  - SenderKeyChain: Chain Key ratchet (H1/H2 分裂)
  - GroupSession: 管理组成员 + 密钥状态

Phase 3: 群组生命周期
  - 创建群组 (生成 group_key + 封装给成员)
  - 邀请成员 (用已有成员的 HybridKem 封装新 group_key)
  - 移除成员 (re-key: 新 group_key 仅分发给剩余成员)
  - 定期轮换 (可选，每 N 条消息或每 T 秒)

Phase 4: Gossipsub 集成
  - 消息类型扩展: GroupSetup / GroupReKey / GroupMessage
  - Envelope payload 承载群组消息
  - Welcome/ReKey 通过 direct message 点对点投递

Phase 5: 消息加密
  - GroupMessage: AEAD.Seal(sender_key_chain.current(), plaintext)
  - 每发送者独立 Chain Key ratchet
  - 消息签名 (Ed25519 + ML-DSA-65 hybrid)

Phase 6: 集成测试
  - 2-node 群组加密通信
  - 3-node 群组: 加入/离开/re-key
  - 消息去重与现有 LRU dedup 兼容

Phase 7: CLI 群组命令
  - /group create, /group invite <peer>, /group list
  - /group leave, /group members
```

### 未来升级路径

```
v0.2.0:  Sender Key + HybridKem (本次实现)
    ↓
v0.2.1:  mKEM 优化 (带宽 9× 减少, 替换 N×HybridKem 为单次 mKEM)
    ↓
v0.3.0:  可选 MLS 集成 (OpenMLS PQC 合并后, 完整 FS/PCS)
```

## 8. libp2p / Gossipsub 集成

**libp2p 没有内置 MLS 或群组加密支持**，也没有相关计划。

Gossipsub 集成方式:
```
Topic: "pqnodium-group-v1"
  → Envelope { payload: GroupMessage | GroupSetup | GroupReKey }
  → 群组消息: AEAD 加密后的密文
  → 管理消息: GroupSetup/ReKey (含加密的组密钥)
```

管理消息的投递策略:
- **GroupSetup**: 创建者 → Gossipsub 广播 (所有成员可见, 各自解封自己的密钥)
- **GroupReKey**: 任意成员 → Gossipsub 广播
- **成员邀请**: 点对点 (libp2p direct message)，避免泄露组密钥给非成员
